# Email Confirmation + Optional 2FA

- Started: 2026-05-06
- Status: planned (Phase 1 = email confirmation, targeting v1.0.4)
- Last updated: 2026-05-06

## Context

Asked by user (2026-05-06):

> "if you create an account on melo, you should have to do a multi
> factor authentication and get a code sent to your email account."

Today, signup is email + password. The Phase 1 backend rollout in
April flipped Supabase auth's "Confirm email" setting **off** so
testing was friction-free. New accounts are usable immediately, with
no proof the user actually owns the email address.

That's fine for a closed alpha. It's not fine now that the app is
in the App Store with real users:

- Signup spam — anyone can create accounts with throwaway typos
  (`aiden@gmial.com`) that lock out the real owner if they later
  try to sign up
- No basis for "contacts discoverable by email" feature in
  `2026-05-06-contacts-buddy-discovery.md` — discovery requires the
  email to be proven, otherwise an attacker can claim
  `someone@gmail.com` and harvest matches
- Account recovery via email is meaningless if the email was never
  verified
- Standard floor for any social-adjacent app

This initiative does two things, in order:

1. **Phase 1 — Email confirmation on signup.** The thing the user
   asked for. Sends a 6-digit OTP to the email, user enters it,
   account becomes active. One-time, on signup only.
2. **Phase 2 — Optional TOTP 2FA.** Power-user feature in Settings.
   Authenticator app (Google Authenticator / 1Password / Authy).
   Required on every login if enabled. Ships separately, no urgency.

Phase 1 is ~1 day of work. Phase 2 is ~3 days. Treat them as
independent.

## Plan — Phase 1 (Email confirmation)

### Supabase auth config

In Supabase Dashboard → Authentication → Providers → Email:

1. Enable **"Confirm email"**
2. Switch confirmation method from magic-link redirect → **OTP code**
   (6-digit code in the email body, no link)
3. Customize the email template:
   - Subject: `Your Melo confirmation code`
   - Body: branded, plain code, copy-paste friendly
   - From: `Melo <hello@melo.show>` (need DNS verification on
     melo.show first — SPF + DKIM)
4. Set "Site URL" to `https://melo.show` (already is) — used as the
   reset-password redirect base

### Client side (`SignUp.jsx` flow)

Current flow:
```
[Sign Up form] → supabase.auth.signUp({email, password}) → Onboarding
```

New flow:
```
[Sign Up form] → supabase.auth.signUp({email, password})
              → "Check your email for a 6-digit code" screen
              → User enters code
              → supabase.auth.verifyOtp({email, token, type: 'signup'})
              → Onboarding
```

Three new pieces:

- **`SignUp.jsx`**: after `signUp()` succeeds, advance internal step
  state to `'awaiting_otp'` instead of waiting for session
- **`OtpEntry.jsx`** (new component): 6 single-character boxes,
  auto-advance on type, paste-to-fill, "Resend code" link with 60s
  cooldown, "Use a different email" escape hatch
- **`SignIn.jsx`**: handle the case where a user with an unconfirmed
  account tries to sign in — surface "Confirm your email first" and
  re-route to OtpEntry

Resend uses `supabase.auth.resend({type: 'signup', email})`.

### Edge case: users who signed up before this lands

Existing accounts in Phase 1 alpha never confirmed their email.
Options:

- **A. Grandfather them.** Existing users keep working. Confirmation
  is enforced for new signups only.
- **B. Force re-confirm.** On next sign-in, prompt "Confirm your email
  to continue" with a one-shot OTP.

Option **B** is the right answer for a social product that's about to
ship contacts/friends features — but only after we've notified the
existing user base via email. Schedule the cutover for v1.1 or later,
not v1.0.4.

### Rate limiting

Supabase enforces a default 5 emails/hour per address. That's enough
for normal flow. Bump if we see failures in practice.

## Plan — Phase 2 (Optional TOTP 2FA)

Out of scope for v1.0.4. Captured here so the initiative is whole.

### Schema

`auth.mfa_factors` is built into Supabase. No app-side migration
needed.

### Settings UI

New "Two-Factor Authentication" card in Settings → Account:

```
┌─────────────────────────────────────┐
│  Two-Factor Authentication          │
│  ───────────────────────────────    │
│  Add an extra step to sign-ins.     │
│                                     │
│  Status: Off                        │
│  [ Set up with authenticator app ]  │
└─────────────────────────────────────┘
```

Setup flow:
1. `supabase.auth.mfa.enroll({factorType: 'totp'})` → returns QR code
2. User scans with authenticator app
3. User enters first 6-digit code → `mfa.challenge` + `mfa.verify`
4. Show 8 backup codes, one-time, with "Save these somewhere safe"
   (for recovery if device is lost)

Sign-in flow:
1. Email + password as today
2. If the user has TOTP enrolled, the session is `aal1` (single-
   factor) instead of `aal2` (full)
3. Block all data fetches at `aal1` — show TOTP entry instead
4. `mfa.verify` upgrades to `aal2`, normal app loads

### Recovery

- Backup codes (one-time use, 8 generated at enroll)
- "Lost your device" flow: email a 24-hour magic-link that, when
  clicked, removes the TOTP factor (security trade-off: user must
  still have email access, which is why Phase 1 email confirmation
  is the prerequisite)

## Phasing

- **Phase 1 — Email confirmation, v1.0.4** (~1 day):
  - Flip Supabase setting on
  - Customize email template + verify melo.show DNS
  - Build OtpEntry component
  - Wire into SignUp flow
  - Update SignIn to surface "confirm email first" error
- **Phase 2 — Optional TOTP, v1.2 or later** (~3 days):
  - Settings UI for enroll + remove
  - Sign-in interceptor for `aal1` → TOTP gate
  - Backup code generation + display + storage
  - "Lost device" recovery flow

## Open questions / follow-ups

- **DNS for melo.show email.** Sending from `hello@melo.show` requires
  SPF + DKIM records. Supabase's default sender (`noreply@mail.app.
  supabase.io`) works out of the box but looks like a phishing email.
  Track in `2026-05-01-llc-formation.md` follow-ups (LLC = the entity
  that owns the domain DNS).
- **Existing-user re-confirm cutover.** Pick a target version and
  send a heads-up email 7 days before. v1.1 is the earliest sensible
  slot.
- **iOS keyboard suggestions.** OTP codes from email auto-suggest into
  the keyboard bar on iOS 17+ if the email subject contains `code` —
  ours does. Confirm autofill works in TestFlight before claiming
  feature complete.
- **Cross-link with `2026-05-06-contacts-buddy-discovery.md`** — that
  initiative requires verified email before discoverability is
  enabled.
- **Cross-link with `2026-04-17-backend-and-social.md`** — this is
  effectively Phase 1.5 of that initiative (auth hardening that was
  deferred during alpha).
- **Apple Sign In + Google Sign In** as alternatives — the user
  asked for email MFA specifically. SSO (which has its own MFA
  built into the identity provider) is a separate feature and not
  in scope here. Worth a future initiative if churn data shows
  signup friction.
