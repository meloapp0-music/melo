# Dark Mode

- Started: 2026-05-12
- Status: planned (v1.1 headline, ~1-1.5 weeks)
- Last updated: 2026-05-12

## Context

Melo today renders in a single cream + brown palette (`--cream`,
`--brown-deep`, `--brown-muted`). Per `CLAUDE.md`, the design language
is "premium iOS-native, frosted glass + gradient accents" — which the
current cream palette does well in daylight contexts.

Two forcing functions push this to v1.1:

1. **Marketing materials show dark mode that doesn't exist.** The
   carousel + the single-image posts (Log every show / See your stats
   / Your year in music / Track every song) all render the iPhone
   screens in dark mode. They look fantastic. They're also
   misleading — when a user downloads, they get cream. The gap will
   compound as more marketing assets stack up.

2. **Concerts happen at night.** Pulling out a screen in a venue +
   logging a show with a bright cream UI is genuinely worse UX than
   a dark UI. Apple Music, Spotify, Bandsintown all default to dark
   for the same reason. Industry pattern.

Bringing dark mode forward to v1.1 closes the marketing gap AND
matches the actual context users open the app in.

## Plan

This is a CSS + design refactor, not a code-architecture one. The
component layer doesn't change — only the colors do.

### A. CSS variable audit + token system

Today `App.css` has the design tokens at the top of the file
(`:root` block with `--cream`, `--brown-deep`, etc.). Some components
use these tokens directly; others hardcode hex values from those
same colors. The audit:

1. Find every hex value in `App.css` (~6500 lines) — there are
   likely 50-100 hardcoded values
2. Map each to a semantic token: `--bg-primary`, `--bg-secondary`,
   `--text-primary`, `--text-secondary`, `--border`, `--accent`,
   `--surface-elevated`, etc.
3. Replace hardcoded hex with `var(--...)` references
4. Same audit in `src/web/components/KineticVibe.jsx` CSS (the
   ~480 lines of kinetic typography styles — many gradients there
   are intentionally cross-mode, but the surface backgrounds need
   tokenizing)

Effort: ~half day for the audit, ~1 day for the find-replace pass.

### B. Design the dark palette

Match the energy of the marketing mockups. From the carousel + the
"Wrapped 03" image, the dark palette is roughly:

| Token | Cream value | Dark value |
|---|---|---|
| `--bg-primary` | `#FBF6EE` | `#0A0F1C` |
| `--bg-secondary` | `#FFFFFF` | `#13182A` |
| `--surface-elevated` | `#FFFFFF` w/ shadow | `#1A2138` |
| `--text-primary` | `#3A2C20` | `#F4F1EB` |
| `--text-secondary` | `#6B5D54` | rgba(244,241,235,0.7) |
| `--border` | `rgba(0,0,0,0.08)` | `rgba(255,255,255,0.08)` |
| `--accent` | `#E8573A` | `#FF6B45` (slight bump) |
| `--accent-secondary` | `#FF8A4C` | `#FF8A4C` (unchanged) |

The Wrapped screen overlays (Slide bg gradients) already work in
both palettes — they use rgba overlays on top of photos, no
adjustment needed.

KineticVibe gradients (the kinetic typography for vibes +
personalities) are designed against a transparent/photo background
and don't need a dark variant — they already shine on either.

### C. Theme toggle + persistence

New Settings card: **"Theme"** with three options:
- ☀️ Light (current cream)
- 🌙 Dark
- ⚙️ System (follows iOS Setting → Display & Brightness)

Default for new users: **Dark.** Existing users keep whatever they
implicitly had (cream). The toggle lives in `Settings.jsx`.

Persistence:
- Read at app start: `localStorage.getItem('melo.theme')` →
  fallback to `system` → match `prefers-color-scheme`
- Write on toggle: localStorage + new column
  `user_settings.theme_preference` (so the choice syncs across
  devices once a user is signed in)

Migration `0010_user_theme.sql` (number subject to ship-time
ordering):

```sql
alter table public.user_settings
  add column if not exists theme_preference text
    check (theme_preference in ('light', 'dark', 'system'))
    default 'system';
```

### D. Component-level QA pass

Some components need dark-mode design love beyond just color flipping:

- **Wrapped slides** — backgrounds use SLIDE_OVERLAYS gradients on
  top of artist photos. The overlays were tuned against the current
  flow; verify they still read well in either mode.
- **ShowDetail** — the hero photo + gradient hero treatment. Heavy
  shadow underneath in cream; needs a different shadow color in
  dark.
- **LogShow form inputs** — the cream `.log-input` style is
  light-on-light today; dark mode needs visible field outlines.
- **Kinetic typography slides** — verify each of the 15 vibe motion
  treatments and 14 personality archetypes reads against a dark
  background. The gradients are bright enough that I expect 90%
  work as-is, but Intimate + Folk Poet + Classical Soul (warm cream
  / ivory palettes) might wash out. They'll need a dark-mode
  variant.
- **Map slide tile layer** — Carto Voyager (current) is dark-friendly
  already. Switching to it as the default + adding a light Voyager
  variant for the cream mode might be cleaner than the inverse.

Effort: ~2 days of polishing once the token swap is done.

## Phases within v1.1

1. **Token audit + replacement.** Half day. App still looks
   identical (cream) after this — just refactored.
2. **Add the dark palette tokens + a `[data-theme="dark"]`
   override.** Half day. Now you can toggle via devtools and
   instantly see the whole app in dark.
3. **Settings UI + persistence.** Half day. Real user-facing
   toggle.
4. **QA pass on Wrapped, ShowDetail, LogShow, Profile.** ~2 days.
5. **Update marketing materials.** None needed — the existing dark
   mockups are now accurate.

Total: ~5-7 working days.

## Open questions / follow-ups

- **Default theme for new users.** Lean dark — matches marketing,
  matches industry. But existing cream users should NOT be
  surprised. Migration: on first launch after v1.1 update, show a
  one-time "We added Dark Mode. You're currently on Light. Try it?"
  prompt with a "Don't ask again" option.

- **Per-screen overrides.** Should the animated map slide always be
  dark regardless of theme? It looks better dark. Probably yes —
  some surfaces should pin to dark regardless. Document each.

- **Kinetic gradients that need warm tones.** Folk Poet (warm cream
  / amber), Classical Soul (ivory), Intimate (warm amber) all wash
  out on dark backgrounds. Either: dark-mode-specific variants of
  those classes, OR shift their gradient stops toward more
  saturation in dark mode.

- **Tile layer for the map slide.** Carto Voyager works for both;
  switching to Carto Dark Matter for dark mode is cleaner. Trivial
  swap based on theme prefer.

- **Color contrast / WCAG AA compliance.** Dark backgrounds with
  thin text can fail accessibility checks. Run the palette through
  WebAIM Contrast Checker before shipping. Expected: `--accent` on
  `--bg-primary` should hit 4.5:1 minimum.

- **Cross-link with marketing materials.** Once dark mode ships,
  the carousel + single-image posts become 100% accurate. No
  changes needed to those assets. Win.

- **Cross-link with `2026-05-12-v1-0-5-favorites-and-polish.md`**
  (not yet written) — v1.0.5 is the smaller polish release that
  ships first. Dark Mode is v1.1.
