# v1.0.1 — Post-launch hotfixes

- Started: 2026-04-28
- Status: in-progress (queued; awaiting v1.0 App Store approval before
  archiving Build 5)
- Last updated: 2026-04-28

## Context

v1.0 (Build 4) is currently in App Store review. While verifying the
build in the iOS simulator we hit a real bug: typing **Zeds Dead** (a
DJ duo) in the Log Show flow surfaces the Deezer artist photo correctly
but produces zero rows in the show picker, so the user appears stuck.

Setlist.fm has Zeds Dead listed and returns setlist records, but the
records have empty `sets.set[]` arrays — common for DJ / electronic
acts where the show happened (venue + date are recorded) but no
track-by-track setlist exists. `fetchSetlists` was filtering those out
with `e.sets?.set?.length > 0`, leaving nothing to display.

This same pattern bites for festivals where Setlist.fm has the gig
recorded but no songs (e.g. early bookings of niche acts), and for any
artist whose first few logged shows on Setlist.fm are still pending
moderator song-by-song fills.

## Decision: don't ship during review

Pushing Build 5 right now would invalidate the in-progress App Store
review queue. The bug only affects searches for acts that don't have
song-level Setlist.fm data, and Apple's reviewer is overwhelmingly
likely to test with a mainstream artist that does. Risk of rejection
from this specific bug is low. Better to land it in v1.0.1.

## Plan

### Fix 1 — show empty-setlist results in the Log Show picker

`src/web/api.js` `fetchSetlists`:
- Drop the `s.sets?.set?.length > 0` filter
- Pull up to 30 raw rows so we have headroom after sort
- Sort by `songCount` desc so songful shows surface first
- Slice top 10 — venue/date-only entries still get to the user when
  that's all Setlist.fm offers

This preserves the "rich setlists first" UX for Coldplay-style
mainstream acts while unblocking DJs and edge cases.

### Fix 2 — replace the dead corsproxy.io fallback (still pending)

`src/web/api.js` line 13 falls back to `https://corsproxy.io/?` when
`VITE_API_PROXY_URL` isn't set. Confirmed today that corsproxy.io now
paywalls server-side requests:

```
{ "error": "Server-side requests are not allowed on your plan.
  Upgrade at https://corsproxy.io/pricing/" }
```

Production is unaffected because `.env.local` points at the deployed
`api-proxy` Edge Function. But anyone cloning the repo without env vars
hits the broken fallback. Options for v1.0.1+:
1. Remove the fallback entirely; throw a loud error if env is unset
2. Self-host a tiny Worker that does the same job
3. Document the env var requirement more clearly

Leaning toward option 1 — silent fallbacks to the wrong endpoint hide
real problems.

### Fix 3 — empty-state copy when search truly returns 0

After Fix 1, the only way `fetchSetlists` returns 0 is when Setlist.fm
genuinely has no record of the artist. Today the dropdown just stays
blank. Plan to add a one-liner: "No Setlist.fm matches — fill in the
venue and date yourself." Lives in `src/web/pages/LogShow.jsx`.

### Fix 5 — Setlist.fm shared-key fallback (eliminates user friction)

Family/early users were hitting the per-user-key onboarding as a wall.
The 3-step "go to setlist.fm, register, request a key, paste it back"
flow is too much for non-technical users and was the #1 launch-day
complaint. Path-1 fix from `2026-05-03-setlistfm-friction-reduction`
considerations: ship a server-side fallback so the app works
out-of-the-box for everyone.

**Changes:**

1. **`supabase/functions/setlistfm-proxy/index.ts`** — read a new
   optional `MELO_SETLISTFM_FALLBACK_KEY` env var. Resolution order
   inside the proxy is:
   1. User's own encrypted personal key, if configured
   2. `FALLBACK_KEY` from env, if set
   3. 400 "no key configured" — same as before
2. **`src/web/api.js`** — `fetchSetlists` no longer early-returns when
   no apiKey hint is passed; the proxy decides server-side. Renamed
   the unused parameter to `_apiKey`.
3. **`src/web/pages/LogShow.jsx`** — removed the "no apiKey" early
   return in the search effect, removed the `(isFutureTab || apiKey)`
   gate on the autocomplete dropdown, and removed the inline
   "Connect Setlist.fm — free, takes 30 sec" hint that nudged users
   toward Settings. Setlists now Just Work for everyone.
4. **`src/web/pages/Settings.jsx`** — reframed the Setlist.fm card
   from "Auto-fill real setlists / Connect your free API key" to
   "Bring your own key (advanced) / Auto-fill works out of the box."
   Integrations row label flipped from "API key required" to
   "Active (shared key)" so the connected state isn't a blocker.

**Risks:**

- *Setlist.fm rate limit on a single key:* ~1 req/sec, ~2880/day. Fine
  for week-1 scale; revisit if Melo passes ~500 daily active searches.
  Cache aggressively at the client/Edge level when this gets close.
- *Setlist.fm ToS:* their docs encourage per-user keys for distributed
  apps. Worst-case response if they notice is key revocation, not
  legal action. Many indie apps run shared keys; rarely an issue at
  small scale. Long-term: apply for partner/app-level access.

**Deploy steps (Aidan, ops):**

1. Find or generate a personal Setlist.fm API key at
   https://www.setlist.fm/settings/api
2. `supabase secrets set MELO_SETLISTFM_FALLBACK_KEY=<key>`
3. `supabase functions deploy setlistfm-proxy --no-verify-jwt`
4. Confirm the function logs no errors on first request after deploy
5. (Optional) leave personal key disconnected in Settings to dogfood
   the fallback path before submitting v1.0.1

### Fix 4 — chicken-and-egg buddy bug in Log Show

Discovered on launch day (2026-05-03). The buddy chip section in
`LogShow.jsx` was guarded by `buddies.length > 0`, and the `buddies`
list in `App.jsx` is *derived* from buddy names embedded in past shows:

```js
const buddies = [...new Set(shows.flatMap((s) => s.buddies || []))]
  .map((name) => ({ id: `label:${name}`, name, color: '#E8573A' }));
```

Result: a brand-new user (or anyone who's never tagged a buddy on a
prior show) saw an empty `buddies` array, the section was hidden, and
there was no way to add a first buddy from the Log Show flow. The
Buddies tab itself doesn't have an "add" affordance either (Phase 2
still pending), so this was a real user-blocking dead end on first run.

**Fix:** always render the "Went With" section on the Attended tab,
add a small input + add button above the chips for free-text buddy
names, and render chips for the union of derived buddies and any
newly-typed names in `selBuddies`. New CSS class `.log-buddy-add`
in `src/web/App.css`.

This is a Phase-1-style local fix that keeps buddies as label strings
on each show. Does not touch the Phase 2 friendships table — that's
still part of the backend-and-social initiative.
