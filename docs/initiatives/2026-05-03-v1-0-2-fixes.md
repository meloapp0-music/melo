# v1.0.2 — Post-launch hotfixes (round 2)

- Started: 2026-05-03
- Status: shipped (Build 6 live in App Store as of 2026-05-06)
- Last updated: 2026-05-06

## Context

v1.0.1 (Build 5) was submitted to App Store Connect on 2026-05-03 with
five fixes (Setlist.fm filter, empty-state copy, buddy chicken-and-egg,
shared-key fallback, Settings reframe).

Within minutes of submitting we found another launch-day bug: the
**Onboarding flow tells users "Friends will find you by username. You
can change it later."** but Settings has no UI to actually edit the
username or display name. The backend (`updateMyProfile` +
`checkUsernameAvailable` in `src/web/lib/db/profiles.js`) was always
ready — only the frontend affordance was missing.

Locked in v1.0.2 to bundle this and any other bugs surfaced once
v1.0.1 lands in users' hands.

## Fixes

### Fix 1 — Inline username + display name edit in Settings

`src/web/pages/Settings.jsx`:

- New imports: `useEffect` from React, `checkUsernameAvailable` from
  `lib/db/profiles`, plus `updateProfile` pulled from `useApp()`.
- New state: `editingProfile`, `editDisplayName`, `editUsername`,
  `usernameStatus` (null | 'checking' | 'available' | 'taken' |
  'invalid'), `profileError`, `savingProfile`.
- Debounced username availability check (400ms) that mirrors the
  Onboarding pattern. Skips the check when the typed value matches the
  user's current username (no-op edit case).
- Validation regex `^[a-z0-9_]{3,24}$` matches the backend constraint.
- The Account row gets an "Edit" button that swaps to inline edit form
  with display name input, username input (`@` prefix, lowercase
  enforcement, live availability hint), Save/Cancel.
- `saveProfile` only sends a patch for fields that actually changed.

No backend changes required.

### Fix 2 — "Edit show" button on Show Detail

The `editingShow` flow in `LogShow.jsx` was already fully wired up; it
hydrates every field from an existing show and calls `updateShow` on
save. But the only UI entry point was the "How was [show]?" CTA on
Home for past Going shows. ShowDetail had Compare and Delete buttons
but no Edit, leaving users with no path to fix typos / wrong venue /
missing notes / mis-tagged buddies on already-logged shows.

**Change:** add `setLogEditTarget` to the `useApp()` destructure in
`ShowDetail.jsx`, render an "Edit show" button above the Compare button
that runs `onClose(); setLogEditTarget(show);`. Reuses 100% of the
existing edit machinery — no new state, no new save handler.

### Fix 3 — Add Buddy modal on Buddies page

`pages/Buddies.jsx` was read-only with an empty-state message that
told users to "Add friends when logging shows!" — but logging a new
show isn't always what users want. Most reported the bug as "I want
to add a buddy from the Buddies page" without a path.

Phase 1 architecture stores buddies as label strings inside each
show's `buddies` array (no separate buddy table). So "adding" a
buddy means picking past attended shows to retro-tag them on. Once
any show has the name, the derived `buddies` list in App.jsx
surfaces the buddy everywhere — Buddies page, LogShow chips, etc.

**Change:**
- `+ Add` button anchored to the right of the Buddies-page header
- Modal: name input + checkbox list of attended shows
- Save: bulk-updates each picked show's `buddies` array (deduping
  if the name already exists on a show)
- New `.buddies-add-btn` and `.buddy-add-show-*` CSS classes

This keeps the Phase 1 data model intact. Real user-account-linked
buddy autocomplete is still a Phase 2 / v1.1+ feature (see
`backend-and-social.md`).

## Decision: don't ship until v1.0.1 is approved + stabilized

Rapid v1.0.1 → v1.0.2 resubmits are flagged by Apple's review queue.
Wait for v1.0.1 to be:
1. Approved by App Review
2. Auto-released to the App Store
3. In real users' hands for ~3–5 days

Then bundle v1.0.2 with whatever else has accumulated.
